# dabs recipes

Each `usecases/NN/usecase.md` is **one reason for dabs to exist** — a use case and
the *ideal* flow an expert user would run, from `brew install` onward. These are
design fiction: they describe the CLI we want, not (yet) the CLI we have. When a
recipe uses a flag or command that doesn't exist, that's the point — the recipe
is the spec.

## The one decision every box makes: where does `/work` come from?

A box always has a `/work` directory. The recipes differ mostly in **how that
directory relates to your host** — this is the mount-vs-copy question. There are
two independent axes:

| | **cwd** (this directory) | **worktree** (fresh git branch off HEAD) |
|---|---|---|
| **copy** (snapshot, box owns it) | `dabs claude` | `dabs claude -w` |
| **mount** (live, host sees writes) | `dabs claude -m` | `dabs claude -mw` |

- **copy** — the box gets its own snapshot. The host is never touched. Work done
  inside is lost unless the agent commits/pushes it out. Right for *disposable*
  runs: fleets, dumb-user tests, throwaway experiments.
- **mount** — a live bind; the box's writes land on the host immediately. Right
  for *iterating*: you and the agent edit the same files.
- **cwd** — `.` as-is. Simple, but a mounted agent edits your actual working tree.
- **worktree** — a new branch checked out under `~/.dabs/worktrees/`. The agent
  works in isolation; your real working tree is untouched; you review the branch
  after. `-mw` (mounted worktree) is the sweet spot for handing off a feature.

Anything a flag can express, a **`dabs.json`** can express declaratively — that's
the route when a box is a reusable artifact (a test rig, a fleet member) rather
than a one-off from your shell. Flags are for humans at a prompt; `dabs.json` is
for things you run 30 times.

## Command surface these recipes assume

```
dabs auth <harness>                 # log in once; credential lives in a host vault
dabs claude [-m|-w|-mw] [prompt]    # start a Claude agent box (source strategy above)
    --name <n>                      #   give it a stable name (else autogenerated)
    --detach                        #   leave it running in the background
dabs up <dir|dabs.json>             # start a box from a manifest
    --replicas <n>                  #   start n identical boxes (a fleet)
    --collect <dir>                 #   gather each box's /out into <dir>/<instance>
dabs ls [--json]                    # list boxes/agents: name, status, branch, driver
dabs attach <name>                  # reattach to a detached agent's live session
dabs logs <name> [-f]               # print (or follow) an agent's transcript
dabs send <name> "<msg>"            # deliver one message into an agent, non-interactively
dabs tail [--all] [--json]          # stream output from one/all agents
dabs run <name> <shell…>            # run a shell command line inside a box (exec for exact argv)
dabs down <name|--all>             # stop + remove
dabs serve [--port <p>]             # expose the above as a local API for a UI/relay
```

Every read command takes `--json` so a script or UI can drive dabs as a backend.

## Index

- `01` — 30 dumb-user tests: a fleet of Haikus against a product (`dabs.json`, copy)
- `02` — a Mac dev hands a feature to a work agent (`-mw`, worktree)
